Remote Job Auto-Apply Pipeline

A structured-output LLM pipeline for intelligent job discovery and application
Mini-Project · Option 1: Structured-Output Workflow  |  github.com/jaimalharsk/dev-lab

Problem

Job seekers spend hours manually reading postings, tailoring resumes, and writing cover letters — most of which go to listings that are already closed or a poor fit. Unstructured LLM outputs (plain text responses) make automation unreliable: a scorer that returns "looks like a 7 or 8" cannot be used in a pipeline.

Solution

A multi-step Python pipeline where every AI output is a validated Pydantic model. The pipeline scrapes jobs, checks posting liveness before spending tokens, scores fit across four dimensions, and generates tailored materials — all with schema enforcement at each stage.

Structured Output Schemas

Pipeline Step Schema Key Validated Fields
Job Scoring JobRelevanceResult role_match, level_fit, growth_potential, remote_alignment (each 0–1) → computed score 1–10
Cover Letter CoverLetter opening, body, closing (enforces section structure)
Resume Tailoring TailoredResume summary, skills[], experience[], education[] (ATS-structured)
Liveness Check LivenessResult status: "active" | "expired" | "uncertain", reason

Scoring Model

The LLM returns four sub-scores. The composite is computed deterministically in code — not by the model:

score = round((role_match × 0.45 + level_fit × 0.30 + growth_potential × 0.15 + remote_alignment × 0.10) × 10)

If any sub-score is outside [0.0, 1.0], Pydantic raises a ValidationError before the value is used anywhere.

Technical Architecture

Tech Stack

  • Python 3.11+ — pipeline orchestration
  • OpenAI API — beta.chat.completions.parse() for structured output
  • Pydantic v2 — schema definition and validation
  • SQLAlchemy + SQLite — job and application persistence
  • Requests + BeautifulSoup — scraping and liveness checks
  • Playwright — form submission automation
  • Streamlit — local tracking dashboard

Project Structure

  • ai/schemas.py — all output models
  • ai/scorer.py — multi-dimensional scoring
  • ai/resume_tailor.py — structured resume
  • ai/cover_letter.py — structured cover letter
  • scrapers/liveness.py — posting freshness check
  • db.py — Job + Application models
  • app.py — pipeline orchestration

Key Design Decisions

DecisionRationale
Weights in code, not in prompt The LLM scores dimensions; arithmetic is deterministic Python. Weights can be tuned without changing any prompt.
Liveness check before scoring Pattern matching on HTTP response is free. Scoring via LLM costs tokens. Checking order matters.
@computed_field for composite score The integer score is derived from sub-scores at validation time — it cannot be set to an arbitrary value independently.
Structured cover letter sections Returning opening/body/closing separately allows post-processing (e.g. swapping closings per company) without re-calling the model.

Safety and Compliance

  • dry_run=True by default — forms are filled but never submitted without explicit opt-in
  • Daily application cap via MAX_APPLICATIONS_PER_DAY environment variable
  • Scraping uses only public pages and ATS APIs — no session/login scraping
  • All personal data (resume, profile) stays local — sent only to the configured OpenAI endpoint

What This Demonstrates

A real multi-step LLM pipeline where every stage produces validated structured output. The pipeline fails loudly on bad model responses rather than propagating garbage silently downstream. Schema design, weighted scoring, and output serialization are all first-class concerns — not afterthoughts.

Limitations and Next Steps

Current limitations

  • Liveness check is pattern-based — does not use a headless browser
  • Scrapers for LinkedIn require rate-limit care
  • Single-threaded — one job scored at a time

Possible extensions

  • Parallel scoring with async OpenAI calls
  • Interview prep (STAR story generation per job)
  • Provider abstraction (Claude / Gemini / local models)
